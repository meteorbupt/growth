# 如何做好技术规划

> 系统化的技术规划方法论，帮助团队制定并落地技术战略

## 背景/场景

技术规划是技术团队管理者的核心职责之一。一个良好的技术规划不仅能提高团队的执行效率，还能确保技术方向与业务目标一致，促进团队成员成长。然而，很多团队在实践中常遇到以下挑战：

- 规划难以落地，经常半途而废
- 技术选型决策缺乏系统性方法
- 外部变化快，规划很快就过时
- 团队成员理解不一致，执行方向偏离

本文结合实践经验和方法论，提供一套可操作的技术规划框架。

---

## 核心内容

### 一、技术规划的定义与层次

技术规划可以分为三个层次：

| 层次 | 说明 | 时间跨度 | 示例 |
|------|------|----------|------|
| **战略层** | 公司/部门级技术方向 | 1-3年 | 技术栈演进、基础设施建设规划 |
| **战术层** | 项目级技术方案 | 3-12月 | 具体项目的架构设计、技术选型 |
| **执行层** | 任务级技术实现 | 1-3月 | 具体功能的实现方案、技术细节 |

### 二、技术规划的核心步骤

#### 步骤1：定义好问题

这是技术规划最关键的第一步。**优秀的人眼中都是活，但在战略决策场景下，你需要找到那个"打蛇打七寸的点"**。

**决策原则排序：**
1. **业务价值 > 范围可控 > 技术深度**

#### 1.1 业务价值评估

**关键原则：有没有业务价值不是你说了算，而是你的客户说了算。**

评估方法：
- 说这句话的人到底花了多少精力跟客户在一起？
- 说这句话的人之前是否有过让客户惊艳的经历？

**表述对比示例：**

| 技术表述（错误） | 业务表述（正确） |
|-----------------|-----------------|
| 把所有应用迁移到Kubernetes | 提升测试环境可靠性，减少因环境问题导致的测试失败率50% |
| 用新的IPAM替换RAS | 彻底杜绝IP分配冲突问题，消除因IP冲突导致的线上故障 |
| 实现基于Envoy的服务网格 | 提升服务间通信的安全性，实现细粒度的权限控制 |

**要点：技术表述关注手段，业务表述关注目的和价值。**

#### 1.2 范围可控评估

**关键原则：在公司做事，需要按阶段交付业务价值。交付时间拖得越长，失败风险越大。**

评估框架：

```
依赖分析矩阵：
┌─────────────┬────────────┬─────────────┐
│   依赖项    │ 重要程度   │ 控制力强度  │
├─────────────┼────────────┼─────────────┤
│ 依赖1       │ 高/中/低   │ 强/中/弱    │
│ 依赖2       │ 高/中/低   │ 强/中/弱    │
│ 依赖3       │ 高/中/低   │ 强/中/弱    │
└─────────────┴────────────┴─────────────┘
```

**决策建议：**
- 对重要依赖且控制力弱的，要么获得领导支持，要么重新审视技术决策以去除依赖
- 如果都搞不定，反思这个问题是否应该你来解决

#### 1.3 技术深度评估

虽然排在最后，但这是最棘手的问题。

**两大现实挑战：**
1. 团队成员关心自己的市场竞争力，技术深度不足会影响积极性
2. 不奔着市面上很火的技术方向走，很难招到高质量人才

**技术深度决策四层框架：**

```
第一层：业务价值优先（技术必须服务于业务）
    ↓
第二层：首选公司内长期无法解决的棘手问题
    ↓
第三层：看看有没有新技术趋势可以乘数效应解决问题
    ↓
第四层：克制贪念，控制作战范围，小步迭代
```

---

#### 步骤2：找到关键的人

技术决策依赖领域内专家，作为管理者需要：

**内部寻找：**
- 识别有兴趣、有潜力的部门内人才
- 制定培养计划

**外部寻找：**
- 动用一切人脉去寻找外部高手咨询
- 甚至直接劝高手加入

**关键点：找到的是领域内专家，而不是通才。**

---

#### 步骤3：理顺关键人员

即使定义了很好的问题，找到了关键专家，也不一定能顺利启动项目。

**核心原则：不要老想着颠覆别人，把相关的人给理顺。**

**实用技巧：**
> 把有可能提反对意见的意见领袖变成你的盟友，帮着你去说服其他人。具体方法就是把问题抛给他，让他参与进来帮你一起想办法，这样他会觉得最终的解决方案有他相当大的贡献。

**为什么要这样做：**
- 避免后续立项遇到阻力
- 意见领袖能提供技术决策参考
- 增加决策透明度和团队信任

---

#### 步骤4：把握技术趋势

**技术趋势时间线：**

```
领域顶级会议论文 → 业界前沿技术公司文章 → 开源项目
    （最早）                  （中间）              （较晚）
```

**越早把握趋势，就有越多的时间来准备你要发起的项目。**

**实战案例：**
- Kubernetes：谷歌2014年开源，2015年发布BORG论文
- ClickHouse：2016年开源，对OLAP分析影响巨大
- Spanner：谷歌2012年发表论文
- Haystack：Facebook 2010年发表论文

### 三、技术选型决策框架

技术选型是技术规划的重要组成部分，可以使用**层次分析法（AHP）**进行系统化决策。

#### 3.1 AHP决策步骤

```
1. 构建决策层次结构
   - 定义总体目标
   - 确定评估标准
   - 列出备选方案

2. 建立判断矩阵
   - 对每个标准进行两两比较
   - 使用1-9标度法赋予相对重要性

3. 计算权重向量
   - 对判断矩阵进行归一化
   - 计算每行的平均值，得到权重向量

4. 一致性检验
   - 计算一致性比率（CR）
   - 确保 CR < 0.1，否则需要重新调整

5. 计算总排序向量
   - 对每个备选方案在各个标准下进行评分
   - 将评分与权重相乘，得到总分

6. 选择最优方案
   - 比较总分，选择得分最高的方案
```

#### 3.2 技术选型评估维度

推荐从以下维度进行评估：

| 维度 | 说明 | 评估要点 |
|------|------|----------|
| **业务契合度** | 是否满足业务需求 | 功能覆盖、扩展性、业务场景适配 |
| **技术成熟度** | 技术稳定性与可靠性 | 社区活跃度、版本稳定性、生产案例 |
| **团队能力** | 团队学习与掌握成本 | 学习曲线、现有技能匹配、培训成本 |
| **生态支持** | 第三方支持与集成能力 | 文档质量、第三方库、工具链 |
| **长期维护** | 可持续性与演进能力 | 商业支持、技术路线、升级成本 |
| **成本效益** | 投入产出比 | 开发成本、运维成本、时间成本 |

### 四、技术规划的落地执行

#### 4.1 OKR与技术规划结合

**OKR核心思想：全员聚焦，保证资源为战略目标做贡献。**

**关系链：**
```
产品战略 → OKR制定 → KPI选择 → 战略落地反馈
```

**技术OKR制定原则：**
- Objective（目标）：明确的业务价值导向
- Key Results（关键结果）：可量化的技术指标

**示例：**
```
O: 提升平台稳定性，保障核心业务连续性
KR1: 核心服务可用性从99.5%提升到99.9%
KR2: 平均故障恢复时间从30分钟降低到10分钟
KR3: 完成80%核心服务的监控告警体系建设
```

#### 4.2 技术路线图制定

**技术路线图三要素：**
- **时间轴**：里程碑节点和交付时间
- **技术栈演进**：从现状到目标的演进路径
- **依赖关系**：各技术组件之间的依赖关系

**技术路线图模板：**

```
季度规划视图：
┌───────────┬───────────┬───────────┬───────────┐
│   Q1      │   Q2      │   Q3      │   Q4      │
├───────────┼───────────┼───────────┼───────────┤
│ 基础调研  │ 原型验证  │ 小范围试点│ 全面推广  │
│ 技术选型  │ 架构设计  │ 系统开发  │ 系统优化  │
│ 团队培训  │ 文档建设  │ 知识沉淀  │ 经验分享  │
└───────────┴───────────┴───────────┴───────────┘
```

#### 4.3 小步迭代原则

**关键原则：克制贪念，控制作战范围，小步迭代。**

**实施要点：**
- 第一阶段先跟现有系统集成，逐步过渡到新方案
- 先选取一个核心系统试点，而不是一上来就做全
- 每个阶段都有明确的交付目标和验证标准

### 五、常见问题与解决方案

#### 问题1：规划难以落地

**原因分析：**
- 目标过于宏大，缺乏阶段性里程碑
- 依赖过多，无法控制关键路径
- 资源投入不足，执行力量分散

**解决方案：**
- 拆解为小目标，设置阶段性交付点
- 识别并管理关键依赖，争取领导支持
- 聚焦核心目标，控制作战范围

#### 问题2：技术选型决策争议

**原因分析：**
- 评估标准不清晰，主观性过强
- 团队成员意见分歧，缺乏共识机制
- 缺乏系统化的决策方法

**解决方案：**
- 使用AHP等系统化决策方法
- 提前沟通，让意见领袖参与决策
- 建立技术选型评审机制

#### 问题3：规划跟不上变化

**原因分析：**
- 缺乏趋势研究，决策滞后
- 规划过于刚性，缺乏弹性调整机制
- 外部环境变化超出预期

**解决方案：**
- 建立技术趋势研究机制
- 采用滚动规划，定期调整
- 设置调整阈值和触发条件

#### 问题4：团队成员理解不一致

**原因分析：**
- 规划文档不够清晰
- 缺乏充分的沟通和解释
- 团队成员背景和关注点不同

**解决方案：**
- 使用业务表述而非技术表述
- 组织规划解读会议
- 建立FAQ文档，统一理解

## 实践案例

### 案例：PaaS团队技术规划实践

**背景：**
团队面临多个挑战：
1. 技术栈向Kubernetes迁移，新旧系统双线作战
2. IP分配冲突等顽固问题长期无法解决
3. 公司向安全方向演进，团队是否要加入

**决策过程：**

**步骤1：问题定义**
- 业务价值：安全合规能力提升
- 范围可控：聚焦基础架构安全强化能力
- 技术深度：零信任、Istio、Envoy等前沿方向

**步骤2：技术深度决策**
```
第一层：学习PCI DSS、GDPR等安全规范
第二层：解决大规模基础架构安全强化难题
第三层：研究零信任、Beyondcorp、Istio趋势
第四层：第一阶段先用Envoy和现有IAM集成
```

**结果：**
- 明确了安全方向的技术规划
- 小步迭代的实施路径
- 团队成员技术成长方向清晰

## 总结反思

### 关键收获

1. **定义问题比解决问题更重要** - 业务价值优先，范围可控优先
2. **技术决策依赖领域专家** - 找到关键的人，理顺关键人员
3. **把握趋势才能抢占先机** - 论文→文章→开源项目的时间线
4. **小步迭代是落地关键** - 克制贪念，控制作战范围

### 待改进方向

1. 建立更完善的技术趋势研究机制
2. 开发技术选型决策工具，提升决策效率
