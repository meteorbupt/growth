# 如何做好技术规划

> 基于《知行：技术人的管理之路》管理规划框架，结合实践经验的系统化技术规划方法论

## 背景/场景

技术规划是技术团队管理者的核心职责之一。根据《知行：技术人的管理之路》，管理规划是管理者构想的管理理想，是和上级达成的"工作协议"，需要管理者回答：**这个团队打算怎么带？**

然而，很多团队在实践中常遇到以下挑战：
- 规划难以落地，经常半途而废
- 技术选型决策缺乏系统性方法
- 外部变化快，规划很快就过时
- 团队成员理解不一致，执行方向偏离
- 团队定位模糊，价值产出不清晰

本文结合《知行》的管理规划框架和实践经验，提供一套可操作的技术规划方法论。

---

## 核心内容

### 一、管理规划四要素框架

根据《知行：技术人的管理之路》，技术规划（管理规划）需要回答四个核心问题：

```
管理规划四要素：
┌─────────────────────────────────────────┐
│  职能：上级为什么要设立这个团队？        │
│  目标：上级对团队的产出有什么期待？      │
│  团队：上级对团队的发展有什么期待？      │
│  路径：需要向上级申请哪些资源？          │
└─────────────────────────────────────────┘
```

**这四个要素的关系：**
- **职能**是团队定位，决定了团队的价值基础
- **目标**是价值体现，明确了团队要取得什么成果
- **团队**是资源基础，决定了目标实现的能力
- **路径**是达成手段，连接资源与目标

---

### 二、职能定位：明确团队核心价值

#### 2.1 为什么职能定位如此重要

**职能定位回答三个核心问题：**
1. 为什么要设置这个团队？
2. 团队如果持续存在下去，其独特价值是什么？
3. 用什么维度来衡量团队价值的高低？

**职能定位的价值：**

| 价值维度 | 说明 | 关键词 |
|----------|------|--------|
| **团队凝聚力** | 清楚团队做什么，提升认同感和归属感 | 团队认同 |
| **有效激励** | 清楚工作意义和价值，实现自我价值 | 成就感 |
| **主动性提升** | 清晰职责，员工才能主动思考做什么 | 不等不靠 |

**团队职能的两个层次：**

```
职责（下限）：团队生存问题
    ↓ 至少要把工作完成
使命（上限）：团队发展问题
    ↓ 管理者自己的规划与期待
```

**关键洞察：**
- 对有的人来讲，"生存"是第一位的
- 对另外一些人来讲，看不到自我价值，"生存"就失去意义

#### 2.2 如何明确团队职能定位

**三步走方法：**

```
第一步：收集信息
├── 听听上级的期待和要求
├── 问问团队成员对未来的期待
├── 与兄弟团队分清边界
└── 结合自己对领域的理解

第二步：提炼升华
├── 用易于理解的文字简化提炼职能
└── 和行业发展趋势结合，设定期待

第三步：确认和主张
├── 争取得到上级的支持和认可
└── 在适当场合宣传团队职责和使命
```

#### 2.3 实战案例：职能定位表述

**对比示例：**

| 技术表述（模糊） | 职能表述（清晰） |
|-----------------|-----------------|
| 把所有应用迁移到Kubernetes | 提升基础设施稳定性，保障业务连续性，成为公司技术演进的核心支撑团队 |
| 用新的IPAM替换RAS | 彻底杜绝IP分配冲突问题，保障网络资源分配的安全可靠，成为公司基础架构安全的中坚力量 |
| 实现基于Envoy的服务网格 | 提升服务间通信的安全性和可控性，成为公司微服务架构演进的技术引领者 |

**要点：技术表述关注手段，职能表述关注价值和定位。**

### 三、目标制定：明确团队产出期待

#### 3.1 目标的三个维度

目标回答的是"通过什么来体现团队价值"，即团队要取得什么成果。

**三类目标：**

| 目标类型 | 说明 | 特点 | 示例 |
|----------|------|------|------|
| **业务目标**（必选） | 从业务角度要达成的目标 | 外在目标 | 上线新功能、客户量、销售额 |
| **专业目标**（可选） | 从专业能力角度要达成的目标 | 内在目标 | 稳定性、服务容量、处理能力 |
| **团建目标**（可选） | 团队规模、梯度的成长发展 | 体现管理规划完整性 | 人员增长、梯队培养 |

#### 3.2 SMART原则

制定目标参考SMART原则：

```
SMART原则：
├── Specific（明确性）：目标要清晰具体
├── Measurable（可衡量性）：要有量化标准
├── Achievable（可达性）：目标要可实现
├── Relevant（相关性）：与团队职能相关
└── Time-bound（时限性）：要有明确时间点
```

**关键原则：目标尽可能要少。**

一旦目标设置过多，资源就分散了，难以实现目标。

#### 3.3 目标制定实战

**业务价值优先原则：**

在制定技术目标时，遵循决策原则排序：
1. **业务价值 > 范围可控 > 技术深度**

**业务价值评估方法：**
- 有没有业务价值不是你说了算，而是你的客户说了算
- 说这句话的人到底花了多少精力跟客户在一起？
- 说这句话的人之前是否有过让客户惊艳的经历？

**目标表述对比：**

| 技术表述（错误） | 业务表述（正确） |
|-----------------|-----------------|
| 把所有应用迁移到Kubernetes，完成100%迁移指标 | 提升测试环境可靠性，减少因环境问题导致的测试失败率50% |
| 用新的IPAM替换RAS | 彻底杜绝IP分配冲突问题，消除因IP冲突导致的线上故障 |
| 实现基于Envoy的服务网格 | 提升服务间通信的安全性，实现细粒度的权限控制，满足安全合规要求 |

**要点：技术表述关注手段，业务表述关注目的和价值。**

### 四、团队规划：明确资源基础

#### 4.1 团队规划的三个维度

团队是实现目标的人，衡量一个团队有三个方面：

```
团队规划三维度：
┌─────────────────────────────────────┐
│  团队规模：现有多少人，未来新增多少人 │
│  团队分工：负责哪几块业务，人员配置   │
│  团队梯度：能力分布，成熟度和复原力   │
└─────────────────────────────────────┘
```

#### 4.2 团队规模规划

**从资源视角看：**

行业里产品、设计、开发、测试、运维人员有大体的比例，可以参考。

```
典型人员比例参考：
产品 : 设计 : 开发 : 测试 : 运维
 1   :  0.5  :  4   :  1.5  :  0.5
```

**从人才视角看：**

下一个时间节点：
- 重点培养哪些人？
- 给他们提供什么样的平台和空间？
- 有能力提供给他们什么指导和支持？
- 期待他们胜任什么职能和角色？

#### 4.3 团队分工规划

**核心问题：**
- 负责哪几块业务？
- 每块业务配置多少人？
- 人员如何分工？

**分工原则：**
- 清晰的职责边界
- 合理的工作负载
- 明确的责任归属

#### 4.4 团队梯度规划

**梯度的意义：**

梯度成熟的团队不会因为某个核心员工休假或离职而垮掉。

```
理想梯队结构：
┌──────────────────┐
│ 核心骨干（20%）   │ ← 关键决策、技术攻坚
├──────────────────┤
│ 中坚力量（60%）   │ ← 日常工作、稳定产出
├──────────────────┤
│ 成长潜力（20%）   │ ← 学习成长、后备培养
└──────────────────┘
```

### 五、路径规划：明确达成手段

#### 5.1 路径规划的两个核心问题

1. **有哪几条路可走？** 通过哪几项重要工作来实现目标？
2. **需要做什么准备？** 各需要投入多少资源？

#### 5.2 可用的达成手段

```
路径选择矩阵：
┌─────────────┬──────────────┬──────────────┐
│   手段      │ 适用场景      │ 特点         │
├─────────────┼──────────────┼──────────────┤
│ 自研        │ 核心能力建设  │ 可控性强、周期长│
│ 招聘专业人才│ 缺失关键能力  │ 快速补位、成本高│
│ 借调工程师  │ 临时性需求    │ 灵活性好、依赖协调│
│ 寻找合作伙伴│ 能力互补      │ 资源共享、边界管理│
│ 外包        │ 非核心工作    │ 效率高、质量把控│
└─────────────┴──────────────┴──────────────┘
```

#### 5.3 资源清单

资源除了人财物，还需要关注：

```
资源类型：
├── 人力资源：人员数量、能力结构
├── 财力资源：预算、成本控制
├── 物力资源：设备、场地、基础设施
├── 时间资源：截止时间、里程碑节点
├── 信息资源：掌握的内外部信息
└── 权限资源：能够使用的权限范围
```

**关键洞察：时间、信息、权限也是非常重要的资源。**

### 六、范围可控评估：管理依赖风险

**关键原则：在公司做事，需要按阶段交付业务价值。交付时间拖得越长，失败风险越大。**

#### 6.1 依赖分析框架

```
依赖分析矩阵：
┌─────────────┬────────────┬─────────────┐
│   依赖项    │ 重要程度   │ 控制力强度  │
├─────────────┼────────────┼─────────────┤
│ 依赖1       │ 高/中/低   │ 强/中/弱    │
│ 依赖2       │ 高/中/低   │ 强/中/弱    │
│ 依赖3       │ 高/中/低   │ 强/中/弱    │
└─────────────┴────────────┴─────────────┘
```

#### 6.2 决策建议

**对每个依赖评估重要程度和控制力：**

| 控制力 | 重要程度 | 决策建议 |
|--------|----------|----------|
| 强 | 高 | 可以依赖，纳入规划 |
| 中 | 高 | 需要备选方案或领导支持 |
| 弱 | 高 | 必须获得领导支持，否则去除依赖或重新审视问题 |
| 弱 | 低 | 可以接受风险，有替代方案 |

**关键决策：如果重要依赖控制力弱，要么获得领导支持，要么重新审视技术决策以去除依赖。**

### 七、技术选型决策框架

技术选型是技术规划的重要组成部分，可以使用**层次分析法（AHP）**进行系统化决策。

#### 7.1 AHP决策步骤

```
1. 构建决策层次结构
   - 定义总体目标
   - 确定评估标准
   - 列出备选方案

2. 建立判断矩阵
   - 对每个标准进行两两比较
   - 使用1-9标度法赋予相对重要性

3. 计算权重向量
   - 对判断矩阵进行归一化
   - 计算每行的平均值，得到权重向量

4. 一致性检验
   - 计算一致性比率（CR）
   - 确保 CR < 0.1，否则需要重新调整

5. 计算总排序向量
   - 对每个备选方案在各个标准下进行评分
   - 将评分与权重相乘，得到总分

6. 选择最优方案
   - 比较总分，选择得分最高的方案
```

#### 7.2 技术选型评估维度

推荐从以下维度进行评估：

| 维度 | 说明 | 评估要点 |
|------|------|----------|
| **业务契合度** | 是否满足业务需求 | 功能覆盖、扩展性、业务场景适配 |
| **技术成熟度** | 技术稳定性与可靠性 | 社区活跃度、版本稳定性、生产案例 |
| **团队能力** | 团队学习与掌握成本 | 学习曲线、现有技能匹配、培训成本 |
| **生态支持** | 第三方支持与集成能力 | 文档质量、第三方库、工具链 |
| **长期维护** | 可持续性与演进能力 | 商业支持、技术路线、升级成本 |
| **成本效益** | 投入产出比 | 开发成本、运维成本、时间成本 |

### 八、技术规划的落地执行

#### 8.1 OKR与技术规划结合

**OKR核心思想：全员聚焦，保证资源为战略目标做贡献。**

**关系链：**
```
产品战略 → OKR制定 → KPI选择 → 战略落地反馈
```

**技术OKR制定原则：**
- Objective（目标）：明确的业务价值导向
- Key Results（关键结果）：可量化的技术指标

**示例：**
```
O: 提升平台稳定性，保障核心业务连续性
KR1: 核心服务可用性从99.5%提升到99.9%
KR2: 平均故障恢复时间从30分钟降低到10分钟
KR3: 完成80%核心服务的监控告警体系建设
```

#### 8.2 技术路线图制定

**技术路线图三要素：**
- **时间轴**：里程碑节点和交付时间
- **技术栈演进**：从现状到目标的演进路径
- **依赖关系**：各技术组件之间的依赖关系

**技术路线图模板：**

```
季度规划视图：
┌───────────┬───────────┬───────────┬───────────┐
│   Q1      │   Q2      │   Q3      │   Q4      │
├───────────┼───────────┼───────────┼───────────┤
│ 基础调研  │ 原型验证  │ 小范围试点│ 全面推广  │
│ 技术选型  │ 架构设计  │ 系统开发  │ 系统优化  │
│ 团队培训  │ 文档建设  │ 知识沉淀  │ 经验分享  │
└───────────┴───────────┴───────────┴───────────┘
```

#### 8.3 小步迭代原则

**关键原则：克制贪念，控制作战范围，小步迭代。**

**实施要点：**
- 第一阶段先跟现有系统集成，逐步过渡到新方案
- 先选取一个核心系统试点，而不是一上来就做全
- 每个阶段都有明确的交付目标和验证标准

### 九、把握技术趋势：抢占先机

**技术趋势时间线：**

```
领域顶级会议论文 → 业界前沿技术公司文章 → 开源项目
    （最早）                  （中间）              （较晚）
```

**越早把握趋势，就有越多的时间来准备你要发起的项目。**

**实战案例：**
- Kubernetes：谷歌2014年开源，2015年发布BORG论文
- ClickHouse：2016年开源，对OLAP分析影响巨大
- Spanner：谷歌2012年发表论文
- Haystack：Facebook 2010年发表论文

### 十、找到关键的人：借力专家

技术决策依赖领域内专家，作为管理者需要：

**内部寻找：**
- 识别有兴趣、有潜力的部门内人才
- 制定培养计划

**外部寻找：**
- 动用一切人脉去寻找外部高手咨询
- 甚至直接劝高手加入

**关键点：找到的是领域内专家，而不是通才。**

---

### 十一、理顺关键人员：争取支持

即使制定了很好的规划，也不一定能顺利启动项目。

**核心原则：不要老想着颠覆别人，把相关的人给理顺。**

**实用技巧：**
> 把有可能提反对意见的意见领袖变成你的盟友，帮着你去说服其他人。具体方法就是把问题抛给他，让他参与进来帮你一起想办法，这样他会觉得最终的解决方案有他相当大的贡献。

**为什么要这样做：**
- 避免后续立项遇到阻力
- 意见领袖能提供技术决策参考
- 增加决策透明度和团队信任

## 实践案例

### 案例：PaaS团队管理规划实践

**背景：**
团队面临多个挑战：
1. 技术栈向Kubernetes迁移，新旧系统双线作战
2. IP分配冲突等顽固问题长期无法解决
3. 公司向安全方向演进，团队是否要加入

**管理规划四要素分析：**

#### 职能定位
```
职责（下限）：保障现有系统稳定运行
使命（上限）：成为公司基础架构安全的中坚力量
```

#### 目标制定
```
业务目标：满足安全合规要求，支撑公司支付业务发展
专业目标：具备大规模基础架构安全强化能力
团建目标：培养2名安全领域核心骨干
```

#### 团队规划
```
规模：现有8人，新增2人
分工：5人负责现有系统，3人负责安全方向
梯度：培养1名高级工程师，1名中级工程师成为安全领域专家
```

#### 路径规划
```
路径1：自研安全能力（核心工作）
路径2：招聘安全领域专家（快速补位）
路径3：与安全部门合作（能力互补）

资源需求：
- 人员：新增2名有安全背景的工程师
- 时间：第一阶段3个月试点
- 信息：学习PCI DSS、GDPR等安全规范
- 权限：获得安全部门的协作支持
```

**结果：**
- 明确了团队定位和发展方向
- 制定了清晰的阶段性目标
- 小步迭代的实施路径
- 团队成员技术成长方向清晰

## 总结反思

### 关键收获

1. **管理规划四要素是框架核心** - 职能、目标、团队、路径形成完整闭环
2. **职能定位是团队价值基础** - 明确职责（下限）和使命（上限）
3. **目标制定要区分类型** - 业务目标、专业目标、团建目标各有侧重
4. **团队梯度保障复原力** - 成熟团队不会因核心人员变动而垮掉
5. **路径选择有多种手段** - 自研、招聘、借调、合作、外包各有适用场景
6. **定义问题比解决问题更重要** - 业务价值优先，范围可控优先
7. **小步迭代是落地关键** - 克制贪念，控制作战范围

### 待改进方向

1. 建立更完善的技术趋势研究机制
2. 开发管理规划可视化工具，提升规划效率
3. 积累更多实践案例，丰富方法论
