# 技术规划技能

基于《知行：技术人的管理之路》管理规划框架的系统性技术规划文档编写技能。

## 调用时机

当出现以下情况时调用此技能：
- 用户需要创建技术规划文档
- 用户提到"技术规划"、"技术路线图"、"技术战略"
- 用户需要规划团队技术方向
- 用户需要进行技术决策或选型
- 用户询问团队定位和目标设定

## 核心框架：管理规划四要素

本技能基于管理规划必须回答的四个核心问题：

```
管理规划四要素：
┌─────────────────────────────────────────┐
│  职能：上级为什么要设立这个团队？        │
│  目标：上级对这个团队的产出有什么期待？  │
│  团队：上级对这个团队的发展有什么期待？  │
│  路径：需要向上级申请哪些资源？          │
└─────────────────────────────────────────┘
```

**四要素关系：**
- **职能** = 团队定位，决定价值基础
- **目标** = 价值体现，明确成果期待
- **团队** = 资源基础，决定目标实现能力
- **路径** = 达成手段，连接资源与目标

---

## 技能工作流程

### 第一步：明确职能定位（团队定位）

**三个核心问题：**
1. 为什么要设置这个团队？
2. 团队如果持续存在下去，其独特价值是什么？
3. 用什么维度来衡量团队价值的高低？

**团队职能的两个层次：**
```
职责（下限）：团队生存问题
    ↓ 至少要把工作完成
使命（上限）：团队发展问题
    ↓ 管理者自己的规划与期待
```

**职能定位三步法：**
```
第一步：收集信息
├── 听听上级的期待和要求
├── 问问团队成员对未来的期待
├── 与兄弟团队分清边界
└── 结合自己对领域的理解

第二步：提炼升华
├── 用易于理解的文字简化提炼职能
└── 和行业发展趋势结合，设定期待

第三步：确认和主张
├── 争取得到上级的支持和认可
└── 在适当场合宣传团队职责和使命
```

**输出模板：**
```markdown
## 职能定位

**职责（下限）：** [必须要做的事情]
**使命（上限）：** [团队期望成为的样子]

**团队价值陈述：**
[清晰的团队独特价值和定位说明]
```

---

### 第二步：制定目标（产出期待）

**三类目标：**

| 目标类型 | 说明 | 优先级 | 示例 |
|----------|------|--------|------|
| **业务目标**（必选） | 从业务角度要达成的目标 | 外在目标 | 上线新功能、客户量、销售额 |
| **专业目标**（可选） | 从专业能力角度要达成的目标 | 内在目标 | 稳定性、服务容量、处理能力 |
| **团建目标**（可选） | 团队规模、梯度的成长发展 | 体现管理规划完整性 | 人员增长、梯队培养 |

**SMART原则：**
```
SMART原则：
├── Specific（明确性）：目标要清晰具体
├── Measurable（可衡量性）：要有量化标准
├── Achievable（可达性）：目标要可实现
├── Relevant（相关性）：与团队职能相关
└── Time-bound（时限性）：要有明确时间点
```

**关键原则：目标尽可能要少。**

**决策优先级：业务价值 > 范围可控 > 技术深度**

**输出模板：**
```markdown
## 目标制定（SMART原则）

### 业务目标（必选）
[明确的业务目标，包含指标和时间线]

### 专业目标（可选）
[专业能力目标]

### 团建目标（可选）
[团队发展目标]
```

---

### 第三步：规划团队（资源基础）

**三个维度：**

```
团队规划三维度：
┌─────────────────────────────────────┐
│  团队规模：现有多少人，未来新增多少人 │
│  团队分工：负责哪几块业务，人员配置   │
│  团队梯度：能力分布，成熟度和复原力   │
└─────────────────────────────────────┘
```

**理想梯队结构：**
```
理想梯队结构：
┌──────────────────┐
│ 核心骨干（20%）   │ ← 关键决策、技术攻坚
├──────────────────┤
│ 中坚力量（60%）   │ ← 日常工作、稳定产出
├──────────────────┤
│ 成长潜力（20%）   │ ← 学习成长、后备培养
└──────────────────┘
```

**关键洞察：梯度成熟的团队不会因核心人员离职而垮掉。**

**输出模板：**
```markdown
## 团队规划

### 团队规模
- 现有：[N] 人
- 新增：[N] 人
- 总目标：[N] 人

### 团队分工
[清晰的职责划分和人员配置]

### 团队梯度
[能力分布和发展计划]
```

---

### 第四步：定义路径（达成手段）

**两个核心问题：**
1. 有哪几条路可走？通过哪几项重要工作来实现目标？
2. 需要做什么准备？各需要投入多少资源？

**可选达成手段：**

```
路径选择矩阵：
┌─────────────┬──────────────┬──────────────┐
│   手段      │ 适用场景      │ 特点         │
├─────────────┼──────────────┼──────────────┤
│ 自研        │ 核心能力建设  │ 可控性强、周期长│
│ 招聘专业人才│ 缺失关键能力  │ 快速补位、成本高│
│ 借调工程师  │ 临时性需求    │ 灵活性好、依赖协调│
│ 寻找合作伙伴│ 能力互补      │ 资源共享、边界管理│
│ 外包        │ 非核心工作    │ 效率高、质量把控│
└─────────────┴──────────────┴──────────────┘
```

**资源清单：**
```
资源类型：
├── 人力资源：人员数量、能力结构
├── 财力资源：预算、成本控制
├── 物力资源：设备、场地、基础设施
├── 时间资源：截止时间、里程碑节点
├── 信息资源：掌握的内外部信息
└── 权限资源：能够使用的权限范围
```

**关键洞察：时间、信息、权限也是非常重要的资源。**

**依赖分析框架：**

```
依赖分析矩阵：
┌─────────────┬────────────┬─────────────┐
│   依赖项    │ 重要程度   │ 控制力强度  │
├─────────────┼────────────┼─────────────┤
│ 依赖1       │ 高/中/低   │ 强/中/弱    │
│ 依赖2       │ 高/中/低   │ 强/中/弱    │
│ 依赖3       │ 高/中/低   │ 强/中/弱    │
└─────────────┴────────────┴─────────────┘
```

**决策逻辑：**
- 高重要度 + 强控制力 → 可以依赖，纳入规划
- 高重要度 + 中控制力 → 需要备选方案或领导支持
- 高重要度 + 弱控制力 → 必须获得领导支持，否则去除依赖
- 低重要度 + 弱控制力 → 可接受风险，有替代方案

**输出模板：**
```markdown
## 路径规划

### 可选路径
[可选手段列表和关键工作项]

### 资源需求
- 人员：[具体需求]
- 预算：[财务需求]
- 时间：[时间线和里程碑]
- 信息：[信息需求]
- 权限：[权限需求]

### 依赖分析
[关键依赖和控制力分析]
```

## 技术选型决策框架

当技术规划包含技术选型时，使用**层次分析法（AHP）**。

**评估维度：**

| 维度 | 说明 | 评估要点 |
|------|------|----------|
| 业务契合度 | 是否满足业务需求 | 功能覆盖、扩展性、业务场景适配 |
| 技术成熟度 | 技术稳定性与可靠性 | 社区活跃度、版本稳定性、生产案例 |
| 团队能力 | 团队学习与掌握成本 | 学习曲线、现有技能匹配、培训成本 |
| 生态支持 | 第三方支持与集成能力 | 文档质量、第三方库、工具链 |
| 长期维护 | 可持续性与演进能力 | 商业支持、技术路线、升级成本 |
| 成本效益 | 投入产出比 | 开发成本、运维成本、时间成本 |

**AHP步骤：**
1. 构建决策层次结构（目标→标准→备选方案）
2. 建立判断矩阵（两两比较，1-9标度）
3. 计算权重向量（归一化、求平均）
4. 一致性检验（CR < 0.1）
5. 计算总排序向量
6. 选择最优方案

## 落地执行指南

### 小步迭代原则

**关键原则：克制贪念，控制作战范围，小步迭代。**

**实施要点：**
- 第一阶段先跟现有系统集成，逐步过渡到新方案
- 先选取一个核心系统试点，而不是一上来就做全
- 每个阶段都有明确的交付目标和验证标准

### OKR与技术规划结合

**OKR关系链：**
```
产品战略 → OKR制定 → KPI选择 → 战略落地反馈
```

**技术OKR制定原则：**
- Objective（目标）：明确的业务价值导向
- Key Results（关键结果）：可量化的技术指标

**示例：**
```
O: 提升平台稳定性，保障核心业务连续性
KR1: 核心服务可用性从99.5%提升到99.9%
KR2: 平均故障恢复时间从30分钟降低到10分钟
KR3: 完成80%核心服务的监控告警体系建设
```

## 使用示例

### 示例1：新团队规划

用户："我需要为新成立的数据团队做技术规划"

**响应方法：**
1. 首先明确职能定位：数据团队为什么存在？独特价值是什么？
2. 制定目标：期望什么业务产出？需要什么技术能力？
3. 规划团队：多少人？什么角色？什么梯度？
4. 定义路径：自研还是招聘？需要什么资源？

### 示例2：技术升级规划

用户："我们团队要做技术栈升级，怎么规划？"

**响应方法：**
1. 职能定位不变，但使命提升
2. 目标聚焦：稳定性提升、能力增强
3. 团队规划：培训需求、技能升级计划
4. 路径：分阶段升级、先试点再推广

### 示例3：技术选型

用户："我们需要选择一个新的数据库，应该怎么决策？"

**响应方法：**
1. 使用AHP框架
2. 定义评估维度和权重
3. 系统化对比备选方案
4. 做出决策并说明理由

## 关键原则总结

1. **职能定位是价值基础** - 明确职责（下限）和使命（上限）
2. **目标制定要区分类型** - 业务目标、专业目标、团建目标各有侧重
3. **团队梯度保障复原力** - 成熟团队不会因核心人员离职而垮掉
4. **路径选择有多种手段** - 自研、招聘、借调、合作、外包各有适用场景
5. **业务价值 > 范围可控 > 技术深度** - 决策优先级排序
6. **小步迭代是落地关键** - 控制范围、先试点、验证后再扩展
7. **依赖管理至关重要** - 高重要度+弱控制力=必须获得领导支持
